<task type="auto">
  <name>Task 1: Add cache usage documentation comments to controller and main</name>
  <files>internal/controller/pipeline_controller.go, cmd/main.go</files>
  <action>
Add comments documenting informer cache usage rationale at each read/write operation site.

The audit findings to codify in comments:

1. In `cmd/main.go` at the `mgr.GetClient()` call (line ~205 where PipelineReconciler is created):
   Add a comment above the Client assignment explaining that `mgr.GetClient()` returns a cached client
   backed by the informer cache. All Get() and List() calls through this client read from the
   in-memory cache, not the API server. This is the controller-runtime default and correct pattern
   for production controllers.

2. In `internal/controller/pipeline_controller.go` at the `r.Get()` call in `Reconcile()` (line ~116):
   Add a comment explaining this Get() reads from the informer cache (not direct API server call)
   because the reconciler's Client field is set to mgr.GetClient() which returns a cached reader.
   The informer cache is populated by the watch established in SetupWithManager().

3. In `internal/controller/pipeline_controller.go` at the `r.Update()` call for finalizer addition (line ~134):
   Add a comment noting that Update() (write operations) go directly to the API server, not through cache.
   This is correct - only reads are cached. The subsequent reconcile will see the updated object via the
   informer watch event.

4. In `internal/controller/pipeline_controller.go` at the `r.Update()` call for finalizer removal in reconcileDelete (line ~201):
   Add a similar comment as #3 about writes going to API server.

5. In `internal/controller/pipeline_controller.go` at the `r.Status().Update()` calls (lines ~382 and ~451):
   Add a comment noting that Status().Update() writes directly to the API server status subresource.
   The informer cache is updated asynchronously via the watch. The ObservedGeneration check at the
   top of Reconcile() handles the cache-lag case where the watch event re-triggers reconciliation
   before the cache reflects the status update.

6. In `internal/controller/pipeline_controller.go` at the `SetupWithManager()` function (line ~477):
   Add a comment documenting that `For(&Pipeline{})` establishes the informer watch that populates
   the cache. This is the read side that enables cached Get() calls in Reconcile().

7. Add a package-level comment block at the top of pipeline_controller.go (after the license, before imports)
   with a section titled "Cache Usage Audit" documenting:
   - This controller uses ZERO List() operations in the reconciliation path
   - The single Get() in Reconcile() reads from the informer cache
   - All writes (Update, Status().Update()) go directly to the API server
   - The informer watch in SetupWithManager() keeps the cache current
   - The ObservedGeneration pattern handles cache-lag gracefully

Keep comments concise (1-3 lines each). Use the format:
// Cache: [explanation]

Do NOT change any logic or behavior. This is documentation only.
