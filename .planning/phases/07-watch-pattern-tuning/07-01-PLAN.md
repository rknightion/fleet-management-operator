Audit and document watch pattern configuration using "Watch:" prefix comments (consistent with "Cache:" and "Reconcile:" patterns from phases 5 and 6).

**1. Resync period in cmd/main.go (WATCH-01):**

Add a "Watch:" comment block to the `ctrl.NewManager` call documenting:
- The current configuration uses no explicit SyncPeriod (nil), which means controller-runtime defaults to NO periodic resync
- This is the CORRECT choice for this operator because:
  - The controller uses watch events for Pipeline resources (real-time updates)
  - ObservedGeneration pattern handles cache-lag gracefully
  - Fleet Management collectors poll every 5 minutes independently
  - No periodic resync is needed because we have no external state drift detection requirement (the operator is the sole writer to Fleet Management for these pipelines)
- If periodic resync were needed (e.g., to detect external drift), it should be set to 10-15 minutes (2-3x collector poll interval)

**2. Workqueue rate limiter in pipeline_controller.go SetupWithManager (WATCH-02):**

Add a "Watch:" comment block to the `SetupWithManager` function documenting:
- The current configuration uses controller-runtime default rate limiter (no explicit WithOptions)
- The default rate limiter in controller-runtime v0.23.0 is `workqueue.DefaultTypedControllerRateLimiter` which combines:
  - `ItemExponentialFailureRateLimiter`: base 5ms, max 1000s (16.6 min) -- handles individual item backoff
  - `BucketRateLimiter`: 10 qps, burst 100 -- overall throughput cap
- This is APPROPRIATE for this operator because:
  - Single Pipeline CRD type with expected low-to-moderate volume
  - Fleet Management API has its own rate limiter (3 req/s via golang.org/x/time/rate)
  - The double rate limiting (workqueue + Fleet API client) provides defense in depth
  - MaxConcurrentReconciles defaults to 1, which is correct for serial Fleet API access

**3. Exponential backoff in pipeline_controller.go (WATCH-03):**

Add a "Watch:" comment block near the error return paths in `updateStatusError` and `handleAPIError` documenting:
- Returning `error` from Reconcile triggers controller-runtime exponential backoff via the workqueue rate limiter
- Returning `ctrl.Result{Requeue: true}` (used for status conflicts) requeues with rate limiter but does NOT count as a failure (no exponential increase)
- Returning `ctrl.Result{RequeueAfter: 10s}` (used for 429 rate limit) bypasses the workqueue rate limiter entirely
- Returning `ctrl.Result{}, nil` (used for validation errors) does NOT requeue -- correct because user must fix spec
- The combination of these four return patterns correctly handles all error scenarios

**4. Watch storm prevention in pipeline_controller.go SetupWithManager (WATCH-04):**

Add a "Watch:" comment documenting why this controller has no watch storm risk:
- Only watches Pipeline CRD via `For(&Pipeline{})` -- no Owns(), no Watches() on secondary resources
- Status updates use Status().Update() (not Update()), so they do NOT trigger spec-change watch events
- Finalizer updates DO trigger a watch event, but the ObservedGeneration guard prevents redundant reconciliation after the re-triggered reconcile adds the finalizer and returns
- No external event sources (no channel watches, no generic event handlers)

**5. Package-level Watch Audit summary:**

Add a "Watch Pattern Audit:" comment block at the package level (after the existing Reconcile Loop Audit block) summarizing:
- Resync: Disabled (nil SyncPeriod) -- appropriate for watch-driven controller
- Rate limiter: Default controller-runtime (5ms-1000s exponential + 10qps bucket)
- Backoff: Four return patterns (error, Requeue, RequeueAfter, nil) correctly mapped to error types
- Storm prevention: Single For() watch, Status subresource updates, ObservedGeneration guard

Use the exact "Watch:" prefix for all comments to maintain grep-ability with "Cache:" and "Reconcile:" patterns.
